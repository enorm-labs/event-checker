### 🟠 F-1 (NEW · ✅ FIXED) — Frannz descriptions retain raw Markdown & promo lines

The copilot.events CMS stores Frannz event descriptions as **Markdown**, which the site renders
literally, so scraped descriptions carry raw `[label](url)` link syntax, leading `-` list-item
dashes, and promotional "Tickets im VVK gibt es bei …" lines.

```sql
SELECT count(*) FILTER (WHERE description ~ 'Tickets im VVK')        AS ticket_line,   -- 32
       count(*) FILTER (WHERE description ~ '\]\(')                  AS markdown_link, -- 55
       count(*) FILTER (WHERE description ~ 'shop\.copilot\.events') AS shop           -- 2
FROM event e
         JOIN event_source es ON e.event_source_id = es.id
WHERE es.source_type = 'FRANNZ';
```

32 / 89 descriptions still contain a ticket promo line; 55 / 89 contain markdown-link artifacts. Samples:

```
-Tickets im VVK gibt es bei [www.eventim.de](www.eventim.de) -
… findet ihr auf [frannz.eu](frannz.eu) Die Spiele, die hier nicht …
-Mail an: [reservierung@quiznightshow.de](quiznightshow.de) (Teilnehmeranzahl …
```

**Root cause:** `FrannzOverviewPageScraper.parseDescription` drops only lines whose trimmed text
`startsWith("Tickets im VVK")` — but the real lines begin with a Markdown list dash, e.g.
`-Tickets im VVK …` / `- Tickets im VVK …`, so the filter never matches, and it does nothing about
inline `[text](url)` link syntax or leading `-` bullets elsewhere in the blurb. The `parseDescription`
unit test passed because its fixture event (Tannz im Frannz) happens to have no leading dash.

**Impact:** user-visible noise in a displayed field (not wrong data, so 🟠 not 🔴), but prominent —
many descriptions *lead* with the dangling ticket line.

### 🟠 F-2 (NEW, cross-venue · ✅ FIXED) — "Nachholtermin vom <date>" suffix leaks into artist/title names

A rescheduled-show note the venue appends to the title survives into the title-as-headliner artist name.

```sql
SELECT DISTINCT a.name
FROM artist a
WHERE a.name ~* 'Nachholtermin'
ORDER BY 1;
```

```
Iggi Kelly Nachholtermin vom 28.04.26-           (Astra)
Pohlmann -Nachholtermin vom 10.01.-              (Astra)
The Dear Hunter -Nachholtermin vom 30.09.2025.   (Frannz)
```

**Root cause:** `stripArtistSuffix` (`EventMappingExtensions.kt`) recovers tour/live/anniversary tails
but has no rule for the German `"… -Nachholtermin vom <date>"` (rescheduled) suffix, so the whole
decorated title is minted as the headliner (e.g. `The Dear Hunter -Nachholtermin vom 30.09.2025.`
instead of `The Dear Hunter`). This is the same *class* as the KNOWN reactive-suffix limitation, but
`Nachholtermin` is a recognisable, structural pattern seen at ≥2 venues — worth a real rule rather
than the denylist. The event **title** itself carries the same note (inherent to the source).

### 🟠/🟢 F-3 (NEW, minor) — Frannz "Quiz Night Show" typed OTHER, not QUIZ

```sql
SELECT title, event_type, count(*)
FROM event e
         JOIN event_source es ON e.event_source_id = es.id
WHERE es.source_type = 'FRANNZ'
  AND event_type = 'OTHER'
GROUP BY 1, 2;
-- Quiz Night Show           | OTHER | 5
-- Fußball Weltmeisterschaft | OTHER | 5
```

The 5 "Quiz Night Show" events map to `OTHER` though a `QUIZ` type exists and the title is
unambiguous. **Root cause:** the scraper types Frannz purely from the `event_typ-*` taxonomy class;
Quiz Night has no such class and its source label is literally "Sonstiges" (= OTHER), so mapping to
`OTHER` is *faithful to the source* — but discovery-by-type misses these quizzes. Optional refinement,
not a defect. (The 5 "Fußball Weltmeisterschaft" → OTHER are `event_typ-kinotv` public-viewings and
are correctly `OTHER`.)
